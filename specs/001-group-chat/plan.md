# Implementation Plan: Group Chats

**Branch**: `001-group-chat` | **Date**: 2026-05-04 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/001-group-chat/spec.md`

## Summary

Add a full Group Chat feature to the existing Flutter chat app, following the identical architecture pattern used for single chats: remote data source тЖТ repository тЖТ Cubit (Freezed states) тЖТ layered UI widgets. Three independent Cubits are used to prevent loading-state coupling: `GroupsCubit` (group list stream), `CreateGroupCubit` (group creation), and `SelectedGroupChatCubit` (selected group messages + send). All Firestore access goes through the existing `DataBaseService` abstraction; direct Firestore SDK use is limited to document-ID generation only.

## Technical Context

**Language/Version**: Dart 3.x / Flutter  
**Primary Dependencies**: `flutter_bloc`, `freezed`, `get_it`, `json_serializable`, `cloud_firestore`, `firebase_auth`  
**Storage**: Cloud Firestore (existing project)  
**Testing**: `flutter_test` тАФ unit tests for data layer and Cubit logic  
**Target Platform**: iOS + Android  
**Project Type**: Mobile app  
**Performance Goals**: Group list loads within 2 s; new messages appear within 3 s (real-time stream)  
**Constraints**: DataBaseService abstraction must be used for all Firestore ops; direct SDK only for doc-ID generation  
**Scale/Scope**: Single feature addition; ~22 new files across data + domain + UI layers

### Step 4 тАФ GroupsRemoteDataSource

**File**: `lib/features/groups/data/datasources/groups_remote_data_source.dart`

Abstract class `GroupsRemoteDataSource` with four methods:

```
Stream<List<GroupModel>> getGroups({required String currentUserId})
Future<void> createGroup({required String currentUserId, required String currentUserEmail, required String groupName, required List<String> membersIds, required List<String> membersEmails})
Stream<List<GroupMessageModel>> getGroupMessages({required String groupId})
Future<void> sendGroupMessage({required String groupId, required String senderId, required String senderEmail, required String text})
```

Implementation `GroupsRemoteDataSourceImpl`:

- Constructor: `const GroupsRemoteDataSourceImpl({required DataBaseService dataBaseService})`

**`getGroups` implementation**:
- Uses `_dataBaseService.collectionStream(path: groupsCollection, builder: (data, id) => GroupModel.fromFirestore(id: id, data: data), queryBuilder: (query) => query.where('members', arrayContains: currentUserId))`
- No explicit sort needed (Firestore ordering handled in `sendGroupMessage` which updates `lastMessageTime`; if ordering by lastMessageTime is desired, add `.orderBy('lastMessageTime', descending: true)` in queryBuilder)

**`createGroup` implementation**:
- Generate doc ID: `final groupId = FirebaseFirestore.instance.collection(groupsCollection).doc().id;`  тЖР only allowed direct SDK use
- Build the group data map from all parameters; include the creator in `membersIds` and `membersEmails` if not already present; set `createdAt` to `Timestamp.now()`; set `lastMessage` to `''`; set `lastMessageTime` to `null`; set `admins` to `[currentUserId]`
- Call `_dataBaseService.setData(path: '$groupsCollection/$groupId', data: groupData)`

**`getGroupMessages` implementation**:
- Path: `'$groupsCollection/$groupId/$messagesCollection'`
- Use `_dataBaseService.collectionStream(...)` with `queryBuilder: (query) => query.orderBy('createdAt', descending: false)`
- Builder: `GroupMessageModel.fromFirestore(id: id, data: data)`

**`sendGroupMessage` implementation**:
- Generate message doc ID: `final messageId = FirebaseFirestore.instance.collection(groupsCollection).doc().id;`
- Build message data map: `{id, senderId, senderEmail, text, createdAt: Timestamp.now()}`
- Write message: `_dataBaseService.setData(path: '$groupsCollection/$groupId/$messagesCollection/$messageId', data: messageData)`
- Update group's `lastMessage` and `lastMessageTime`: `_dataBaseService.setData(path: '$groupsCollection/$groupId', data: {'lastMessage': text, 'lastMessageTime': Timestamp.now()}, merge: true)`  тЖР `merge: true` is the default so existing fields are preserved

## Pitfalls to Avoid

| Pitfall | Mitigation |
|---------|-----------|
| Double-subscribing to the groups stream | Guard with `_isListeningToGroups` flag, same as ChatsCubit |
